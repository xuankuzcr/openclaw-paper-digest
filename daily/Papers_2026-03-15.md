### 🦾 UMI & Teaching Devices（1）

#### ⭐ Coarse-Guided Visual Generation via Weighted h-Transform Sampling
- **主题**：UMI & Teaching Devices
- **重点命中**：data collection
- **论文链接**：
- **辅助图示**：https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12057.png
![Coarse-Guided Visual Generation via Weighted h-Transform Sampling](https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12057.png)
- **这篇在做什么**：可以把它理解为：先有一张模糊或低质量参考图，再让预训练扩散模型在生成高清结果时“尽量贴着参考走”。不同于依赖精确下采样公式的方法，它在每一步采样里加一个引导漂移项，把随机生成过程往满足粗条件的方向推。后期噪声变小时再减弱这股推力，避免为了对齐参考而牺牲观感。
- **核心方法**：输入是粗参考（低清/退化观测）与预训练扩散模型中的当前噪声状态表示，目标是生成对应细节样本。关键模块是基于h-transform思想构造的逐步转移修正：在原采样微分方程上叠加漂移函数，实现条件约束下的概率路径重加权。推理时逐时间步迭代采样，并用“噪声水平感知”的权重日程调节引导强度，前期强约束、后期降权以抑制近似误差累积。
- **和以前有什么不同**：相较以往训练自由引导法，它不再强依赖已知的fine-to-coarse前向算子，因此在未知或复杂退化场景更实用。相较简单固定权重或硬约束策略，它通过噪声相关降权机制处理引导误差与生成质量冲突，强调“可控平衡”而非单点最优。整体上从工程技巧上升到随机过程约束视角，理论动机更完整。
- **你能学到什么**：阅读时重点抓三点：h-transform在这里不是抽象数学装饰，而是把“条件生成”转成“转移概率重权”的实现准则。其次要理解漂移项是近似量，误差会随时间步和噪声区间放大，因此日程函数是方法成败核心。最后可思考其与SDE/ODE采样器、classifier-free guidance的关系，便于在你自己的生成管线里替换落地。
- **关键洞见**：核心洞见是：在不重训模型的前提下，最有效的粗引导不是一味加大约束，而是把约束写进每一步随机动力学并随噪声阶段自适应调节。这样既保留预训练扩散模型的先验生成能力，又让结果朝粗参考稳定收敛。换言之，方法把“可控性”和“逼真度”通过时间依赖权重实现了动态折中。

### 📡 Embodied Data Collection Devices（1）

#### ⭐ OmniStream: Mastering Perception, Reconstruction and Action in Continuous Streams
- **主题**：Embodied Data Collection Devices
- **重点命中**：embodied agent
- **论文链接**：
- **辅助图示**：https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12265.png
![OmniStream: Mastering Perception, Reconstruction and Action in Continuous Streams](https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12265.png)
- **这篇在做什么**：作者想解决的是：一个机器人看连续视频时，既要理解语义，又要保持时序因果，还要具备空间几何感。OmniStream把这些能力放进同一模型，通过逐帧处理与缓存历史信息实现实时运行。结果显示即使主干冻结，也能在多类任务上表现稳定，说明表示本身具有较强通用性。
- **核心方法**：输入是连续视频流及相关多模态监督信号，表示层面采用因果时空注意力并结合3D-RoPE，让模型同时编码时间顺序与三维结构线索。关键模块是支持在线推理的持久化KV-cache，使每帧只增量计算而不反复回看全序列。训练上采用多任务联合预训练，耦合静态/时序表征学习、流式几何重建和视觉-语言对齐；推理时可冻结主干并适配下游任务。
- **和以前有什么不同**：相较以往只擅长图像语义或离线视频建模的方法，OmniStream强调严格因果的流式处理，避免依赖全片段回放。相较几何重建与VLM分离的路线，它把语义、时序、空间三类能力放在统一骨干中共同学习。与“单任务专用专家”相比，论文更关注跨任务稳健泛化与可复用性，而非单点极限指标。
- **你能学到什么**：阅读时可重点关注三点：因果注意力如何限制信息泄漏、3D-RoPE如何在时空坐标中注入结构先验、KV-cache如何平衡效率与长期记忆。其次要理解其多任务预训练为何能互补，而不是互相干扰。最后可思考在你自己的场景里，是否也能采用“冻结统一主干+轻量头部”的部署范式。
- **关键洞见**：论文的核心启发是：通用视觉智能的关键不只是更大数据或更深网络，而是把因果时序、空间几何和语义对齐在同一流式表示中联合约束。只要主干学到这种结构化表征，即便不端到端微调，也能支撑多种下游能力。它为具身场景给出了一条“统一骨干优先”的可行工程路径。

### 🎮 4DGS + Diffusion Simulator（2）

#### Mobile-GS: Real-time Gaussian Splatting for Mobile Devices
- **主题**：4DGS + Diffusion Simulator
- **论文链接**：
- **辅助图示**：https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.11531.png
![Mobile-GS: Real-time Gaussian Splatting for Mobile Devices](https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.11531.png)
- **这篇在做什么**：可以把它理解为“给3D高斯渲染做移动端瘦身和加速”。它先绕开最耗时的深度排序与alpha混合瓶颈，用近似但高效的无序渲染思路提速；再用一个神经增强模块补回因无序带来的透明与重叠伪影。最后再通过蒸馏、量化和剪枝，把模型压到手机更容易承载的规模。
- **核心方法**：输入是训练好的3D高斯场景表示，包含几何位置、尺度、透明度与外观属性，并结合视角方向进行渲染。关键模块包括深度感知的顺序无关渲染、视角相关神经增强，以及面向存储的SH蒸馏、神经向量量化和基于贡献度的高斯剪枝。流程上先用无排序管线快速生成结果，再由增强网络修正视角与重叠区域误差，同时在部署前完成表示压缩以兼顾实时性与内存占用。
- **和以前有什么不同**：相较传统3DGS依赖逐像素深度排序与alpha合成的做法，Mobile-GS核心增量是把“必须排序”改成“深度感知的顺序无关”以显著降低移动端计算负担。与只做压缩的轻量化方法不同，它同时处理渲染正确性问题，通过视角相关增强减少无序渲染伪影。并且它将压缩策略与渲染管线协同设计，更贴近真实端侧部署约束。
- **你能学到什么**：阅读这篇可重点学习一个思路：先定位端侧最贵算子，再用可控近似替代，并用学习模块补偿精度损失。第二个收获是移动化不只是参数变小，还要考虑渲染顺序、内存带宽、表示冗余等系统级因素。实践中可借鉴其“加速-补偿-压缩”三段式设计框架来改造其他神经渲染方法。
- **关键洞见**：最关键洞见是：在移动端，严格物理正确的排序合成未必是最优选择，适度放松顺序约束可换来巨大实时性收益。只要再引入面向视角和几何属性的神经补偿，就能在速度与画质之间取得更优平衡。再叠加结构化压缩与剪枝，3DGS才真正具备端侧可部署性。

#### Understanding by Reconstruction: Reversing the Software Development Process for LLM Pretraining
- **主题**：4DGS + Diffusion Simulator
- **论文链接**：
- **辅助图示**：https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.11103.png
![Understanding by Reconstruction: Reversing the Software Development Process for LLM Pretraining](https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.11103.png)
- **这篇在做什么**：作者认为，模型只看成品代码，就像只看成片而没看拍摄过程，难学到复杂决策链。于是他们反向重建“开发轨迹”，让模型学习从需求到实现中间发生了什么。核心直觉是：过程监督比结果监督更能教会模型做长期、可纠错的编程思考。
- **核心方法**：输入是现有代码仓库及其结构信息，如文件层级、模块依赖与代码上下文；表示上构造成多步“代理式开发轨迹”，包含规划、实现、调试和修正的文本/代码链。关键模块包括多智能体仿真器（生成候选轨迹）与基于搜索的CoT优化器（迭代改写推理，使其更能导向真实代码）。训练流程是先合成并筛选高一致性轨迹，再做持续预训练；推理时模型直接利用学到的过程模式完成更长链路任务。
- **和以前有什么不同**：以往代码预训练大多基于静态语料或指令样本，重点在“结果拟合”，较少显式恢复开发过程。该方法把仓库结构约束引入轨迹合成，降低了纯幻想式步骤生成的风险，并用搜索优化提升推理链与真值代码的一致性。增量价值在于把“agentic过程数据”系统化地变成可规模化训练信号。
- **你能学到什么**：可借鉴点一是数据观：高质量过程数据可以通过重建获得，不必完全依赖人工标注。可借鉴点二是训练观：先保证轨迹与代码终态一致，再喂给模型，比只追求语言流畅更关键。实践中应重点评估轨迹真实性、结构约束强度与搜索成本之间的平衡。
- **关键洞见**：真正限制代码大模型上限的，可能不只是参数量或语料规模，而是监督信号缺少“如何到达答案”的中间因果链。把软件工程从终态文本还原为决策过程，能更有效注入长时程推理与自我修正能力。换言之，重建过程本身就是一种更接近智能行为的预训练目标。

#### ⭐ Automatic Generation of High-Performance RL Environments
- **主题**：Other
- **论文链接**：
- **辅助图示**：https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12145.png
![Automatic Generation of High-Performance RL Environments](https://cdn-thumbnails.hf-mirror.com/social-thumbnails/papers/2603.12145.png)
- **这篇在做什么**：论文核心是在说：让代码代理按一套固定配方，把原本慢或未优化的RL环境自动改写成高吞吐实现，同时尽量保持语义一致。作者不是只看速度，而是用分层测试去确认行为是否等价，再用跨后端策略迁移验证没有仿真落差。结果表明，环境工程可以从“手工匠活”变成“可流程化生产”。
- **核心方法**：输入可以是已有环境代码或文本规格，先被组织为统一任务描述与约束表示（规则、接口、可观测量、奖励与终止条件）。关键模块包括通用提示模板、分层验证器（性质测试/交互测试/rollout测试）和迭代修复代理；代理根据失败用例定向改代码。流程上先生成候选高性能实现，再跑分层测试闭环修复，最后做训练吞吐与跨后端策略迁移检查。
- **和以前有什么不同**：相较以往主要依赖专家手写优化内核与长期调参，这里把迁移过程产品化为可复用recipe，强调低成本自动化与可复现步骤。它也不只在单一引擎内做微调，而是覆盖Rust/JAX等不同后端，并在多环境验证语义一致性。新增价值在于把“性能提升”与“等价性证明”并列为一等目标。
- **你能学到什么**：可借鉴之处是先定义严格验证分层，再追求速度；否则容易得到快但不对的模拟器。实践上应准备最小可判定的性质与交互脚本，让代理修复有明确反馈信号，并保留失败轨迹用于回归测试。若用于新环境生成，先把规则规格结构化再代码化，会明显降低后续歧义与返工。
- **关键洞见**：：RL环境优化的瓶颈不只是编译或并行技术，而是缺少“生成—验证—修复”闭环。只要把语义约束显式化并分层检验，代码代理就能在较低成本下稳定产出可训练、可部署的高性能环境。进一步看，当模型规模上升，环境开销占比下降，这种自动化基建价值会更突出。
